线程池：

void (*func)(int) 是 C 语言常用的函数指针。
std::function<void(int)> 是 C++ 的升级版，可以接受几乎所有可调用对象（普通函数、lambda、有捕获的 lambda、仿函数对象、成员函数绑定后的对象等）。
函数指针只能指向“纯函数”，不能保存对象状态（例如成员变量），因为它只能存函数地址。
而 std::function 可以保存对象和状态，所以几乎所有可调用对象都可以接受。
常用示例：

函数签名                std::function 类型       调用方式
void f()                 function<void()>       f()
void f(int)              function<void(int)>    f(10)
int f(int,int)           function<int(int,int)> f(2,3)

线程池中的工作线程 一开始都是阻塞状态，通过条件变量等待任务，避免 CPU 空转。
当有任务提交时，通过 notify_one 或 notify_all 唤醒线程。
被唤醒的线程 先抢锁，检查条件（队列非空或线程池停止）是否满足，再取任务执行。
取完任务后 立即释放锁，任务在没有锁的情况下执行，保证 任务执行阶段完全并发。
加锁只保护共享的任务队列，防止数据竞争。
条件变量wait会临时释放锁，唤醒时再加锁，所以不会因为加锁而死锁或长期阻塞。

这个程序在所有任务结束之后，实际上会进入所有线程都是挂起的状态，直到主线程调用shutdown结束

唤醒：线程被 notify_one / notify_all 或伪唤醒叫醒，它从睡眠态变为“尝试运行”状态。
加锁：线程先重新获取 mutex，确保访问共享资源安全。
判断 predicate：
如果返回 true → 条件满足，线程继续执行 wait 之后的代码（比如取任务、处理任务）
如果返回 false → 条件不满足，线程立刻释放锁，重新回到等待状态（sleep），直到下次唤醒
所以 predicate 决定线程被唤醒后能否真正继续工作，而唤醒本身和 predicate 的结果是两个独立的步骤。
可以理解成：“唤醒是门铃响，predicate 是门锁，只有门锁开了才能进去干活。”

真正的唤醒是指线程在等待 condition_variable 时，有另一个线程调用了 notify_one() 或 notify_all()，然后被操作系统唤醒。
伪唤醒是指线程在没有任何线程调用 notify_one() 或 notify_all()，条件也未满足的情况下，从 wait() 返回。C++ 标准允许“伪唤醒”，并且程序必须防范。



并发bug：
数据竞争：没有锁，没有 atomic，多线程同时修改。目的不是算错一个数字，而是让你看到：程序不崩、代码不报错、结果却“不可信”。
死锁：日志可能停在某一行，程序不退出，CPU 占用可能不高，但服务已经“僵死”
活锁：效果是：两个线程都“很礼貌”，都在让，都在 retry，谁也干不成事。程序在跑，但系统没有进展。

死锁：线程 完全停住，互相等待对方释放资源 → 系统僵死
活锁：线程 一直在动、一直在重试，但 无法推进逻辑 → 系统“空转”

能不能体现出数据竞争，可能还需要根据线程的数量以及CPU核心的关系决定。如果说核心比较多，那么可以支持更多的线程并行发生，也就可能不会发生数据竞争。运行次数也很重要，如果太少，也很难有效果。
我感觉活锁就像是死锁的释放资源后循环的版本



使用select模型：
这种模型在面对多个客户端的时候，可以选择阻塞式accept或者非阻塞式accept。
阻塞式的话，每次只能accept一个客户端，读取发送的信息的时候，也只能读取一句。
非阻塞式的话，可以将准备就绪的全部读入，但是要设置成非阻塞式。


高并发服务器实验简要总结:

一连接一线程模型:
线程数量随连接数线性增长
大量线程阻塞在 read()
频繁上下文切换，CPU 系统态占比高
内存和 Swap 压力大，扩展性差

select 单线程模型:
始终只有 1 个线程
统一等待所有 I/O 事件
上下文切换极少，CPU 利用更高效
内存占用稳定，适合高并发 I/O 场景

高并发服务器的核心目标不是“多跑线程”，而是“减少不必要的上下文切换”。

本质结论:
高并发 ≠ 多线程
I/O 密集场景下，瓶颈在“等待 I/O”，不是算力
I/O 复用用更少的线程完成更多的连接
因此：在高并发 TCP 场景下，select 明显优于一连接一线程模型。

资源使用分析（select vs 一连接一线程）
1️⃣ CPU 使用情况:
一连接一线程:线程数量暴增（1000+）,大量线程频繁被唤醒/挂起,系统态（sy）CPU 占比高,上下文切换成本显著。
select 模型：始终只有 1 个线程，几乎无上下文切换，CPU 使用更集中、更高效，系统态开销明显降低
✅ 结论：select 用更少的 CPU 管理更多连接。

2️⃣ 内存使用情况：
一连接一线程：每个线程占用独立栈空间，线程控制块（TCB）数量巨大，物理内存迅速上涨，容易触发 Swap
select 模型：单线程，无线程栈叠加，内存占用随连接增长缓慢，Swap 使用极少甚至为 0
✅ 结论：select 内存占用更稳定、可控。

3️⃣ Swap 使用情况
一连接一线程：内存压力大，不活跃线程被换出，Swap 使用明显
select 模型：内存压力小，几乎不依赖 Swap
✅ 结论：select 更不容易拖慢系统整体性能。

4️⃣ 负载（Load Average）
一连接一线程：大量 sleeping 线程，Load Average 高，系统调度压力大
select 模型：线程数恒定，Load Average 较低，系统状态更健康

select 模型用更少的 CPU、内存和调度成本，支撑了远多于线程数的并发连接，是 I/O 密集型服务器的更优选择。
