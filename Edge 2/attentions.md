线程池：

void (*func)(int) 是 C 语言常用的函数指针。
std::function<void(int)> 是 C++ 的升级版，可以接受几乎所有可调用对象（普通函数、lambda、有捕获的 lambda、仿函数对象、成员函数绑定后的对象等）。
函数指针只能指向“纯函数”，不能保存对象状态（例如成员变量），因为它只能存函数地址。
而 std::function 可以保存对象和状态，所以几乎所有可调用对象都可以接受。
常用示例：

函数签名                std::function 类型       调用方式
void f()                 function<void()>       f()
void f(int)              function<void(int)>    f(10)
int f(int,int)           function<int(int,int)> f(2,3)

线程池中的工作线程 一开始都是阻塞状态，通过条件变量等待任务，避免 CPU 空转。
当有任务提交时，通过 notify_one 或 notify_all 唤醒线程。
被唤醒的线程 先抢锁，检查条件（队列非空或线程池停止）是否满足，再取任务执行。
取完任务后 立即释放锁，任务在没有锁的情况下执行，保证 任务执行阶段完全并发。
加锁只保护共享的任务队列，防止数据竞争。
条件变量wait会临时释放锁，唤醒时再加锁，所以不会因为加锁而死锁或长期阻塞。

这个程序在所有任务结束之后，实际上会进入所有线程都是挂起的状态，直到主线程调用shutdown结束

唤醒：线程被 notify_one / notify_all 或伪唤醒叫醒，它从睡眠态变为“尝试运行”状态。
加锁：线程先重新获取 mutex，确保访问共享资源安全。
判断 predicate：
如果返回 true → 条件满足，线程继续执行 wait 之后的代码（比如取任务、处理任务）
如果返回 false → 条件不满足，线程立刻释放锁，重新回到等待状态（sleep），直到下次唤醒
所以 predicate 决定线程被唤醒后能否真正继续工作，而唤醒本身和 predicate 的结果是两个独立的步骤。
可以理解成：“唤醒是门铃响，predicate 是门锁，只有门锁开了才能进去干活。”

真正的唤醒是指线程在等待 condition_variable 时，有另一个线程调用了 notify_one() 或 notify_all()，然后被操作系统唤醒。
伪唤醒是指线程在没有任何线程调用 notify_one() 或 notify_all()，条件也未满足的情况下，从 wait() 返回。C++ 标准允许“伪唤醒”，并且程序必须防范。



并发bug：
数据竞争：没有锁，没有 atomic，多线程同时修改。目的不是算错一个数字，而是让你看到：程序不崩、代码不报错、结果却“不可信”。
死锁：日志可能停在某一行，程序不退出，CPU 占用可能不高，但服务已经“僵死”
活锁：效果是：两个线程都“很礼貌”，都在让，都在 retry，谁也干不成事。程序在跑，但系统没有进展。

死锁：线程 完全停住，互相等待对方释放资源 → 系统僵死
活锁：线程 一直在动、一直在重试，但 无法推进逻辑 → 系统“空转”

能不能体现出数据竞争，可能还需要根据线程的数量以及CPU核心的关系决定。如果说核心比较多，那么可以支持更多的线程并行发生，也就可能不会发生数据竞争。运行次数也很重要，如果太少，也很难有效果。
我感觉活锁就像是死锁的释放资源后循环的版本

高并发服务器实验简要总结:

一连接一线程模型:
线程数量随连接数线性增长
大量线程阻塞在 read()
频繁上下文切换，CPU 系统态占比高
内存和 Swap 压力大，扩展性差

select 单线程模型:
始终只有 1 个线程
统一等待所有 I/O 事件
上下文切换极少，CPU 利用更高效
内存占用稳定，适合高并发 I/O 场景

高并发服务器的核心目标不是“多跑线程”，而是“减少不必要的上下文切换”。

本质结论:
高并发 ≠ 多线程
I/O 密集场景下，瓶颈在“等待 I/O”，不是算力
I/O 复用用更少的线程完成更多的连接
因此：在高并发 TCP 场景下，select 明显优于一连接一线程模型。

资源使用分析（select vs 一连接一线程）
1️⃣ CPU 使用情况:
一连接一线程:线程数量暴增（1000+）,大量线程频繁被唤醒/挂起,系统态（sy）CPU 占比高,上下文切换成本显著。
select 模型：始终只有 1 个线程，几乎无上下文切换，CPU 使用更集中、更高效，系统态开销明显降低
✅ 结论：select 用更少的 CPU 管理更多连接。

2️⃣ 内存使用情况：
一连接一线程：每个线程占用独立栈空间，线程控制块（TCB）数量巨大，物理内存迅速上涨，容易触发 Swap
select 模型：单线程，无线程栈叠加，内存占用随连接增长缓慢，Swap 使用极少甚至为 0
✅ 结论：select 内存占用更稳定、可控。

3️⃣ Swap 使用情况
一连接一线程：内存压力大，不活跃线程被换出，Swap 使用明显
select 模型：内存压力小，几乎不依赖 Swap
✅ 结论：select 更不容易拖慢系统整体性能。

4️⃣ 负载（Load Average）
一连接一线程：大量 sleeping 线程，Load Average 高，系统调度压力大
select 模型：线程数恒定，Load Average 较低，系统状态更健康

select 模型用更少的 CPU、内存和调度成本，支撑了远多于线程数的并发连接，是 I/O 密集型服务器的更优选择。


poll模型：
pollfd结构体：
struct pollfd {
    int   fd;        // 要监控的文件描述符
    short events;    // 希望监控的事件类型（输入）
    short revents;   // 实际发生的事件类型（输出，由poll填充）
};
poll 相比 select，在“可扩展性”上更好，但在“高并发性能”上并不一定更好。
poll 相比 select，去除了 fd 数量的硬限制，能够同时处理更多客户端连接；但由于每次都需要线性扫描所有 fd，在高并发但低活跃场景下，CPU 资源利用率并不高。


epoll模型：

int epoll_create1(int flags);
作用：创建一个 epoll 实例，返回一个特殊的文件描述符（epfd），之后所有 epoll 相关操作都基于这个实例。

参数：flags：0：默认；EPOLL_CLOEXEC：当进程调用 exec 系列函数时自动关闭 epfd，防止泄漏到子进程‘

返回值：成功：epoll 实例对应的 fd（整数）  失败：-1，设置 errno

在内核中创建一个 epoll 内部对象。内部保存一个 监听集合（红黑树），记录被监听的 fd；内部有一个 就绪队列，记录已发生事件的 fd
返回一个 fd（epfd）给用户态，用户态后续用它调用 epoll_ctl 或 epoll_wait


int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
作用：操作 epoll 实例的监听集合：增加、修改或删除某个 fd 的事件监听

参数	含义
epfd	epoll_create1 返回的 epoll 实例 fd
op	    操作类型：EPOLL_CTL_ADD / EPOLL_CTL_MOD / EPOLL_CTL_DEL
fd	    要操作的文件描述符（socket）
event	事件描述结构体（epoll_event），包含事件类型 + 用户态标识

内核层面做了什么

根据 op：
EPOLL_CTL_ADD：把 fd 加入监听集合
EPOLL_CTL_MOD：修改 fd 的事件
EPOLL_CTL_DEL：删除 fd
把 epoll_event 保存到内核

epoll_event:
events：你关心的事件类型
data：用户态标识（fd/pointer/id）
建立 fd → epoll 的映射关系,内核用这个映射查找就绪事件

int nready = epoll_wait(epfd, events.data(), events.size(), -1);
作用：等待 epoll 实例中的事件发生

返回值 nready：实际发生事件的数量

参数解释：
epfd → epoll 实例的 fd（epoll_create1 返回的）
events.data() → 用户态数组首地址，用来接收事件信息
events.size() → 用户态数组最大容量，告诉内核最多可以写入多少事件
阻塞时间（毫秒）
-1：无限阻塞，直到至少有一个事件就绪
0：非阻塞检查
>0：等待指定毫秒数

优雅退出：
信号	触发条件
SIGINT	用户按下 Ctrl+C（中断程序）
SIGTERM	系统发送 终止请求，如 kill -TERM pid

优雅退出的核心目标：
优雅退出的本质是 在停止服务时，不丢失任务、不造成资源泄露、不让系统进入不确定状态。具体目标包括：
1、所有正在执行的任务完成
已经被 worker 拿到执行权的任务必须执行完。
避免直接杀线程（std::terminate 或 kill -9）导致任务中途被终止。
2、队列中的任务可控处理
对于尚未执行的任务，要么全部执行，要么根据策略安全丢弃并记录日志。
避免主线程退出导致队列任务丢失。
3、释放资源
内存、文件句柄、socket、互斥锁等都需要安全释放。
线程 join、文件 flush、socket close 都是必要操作。
4、外部信号响应
能够响应系统信号（如 SIGINT Ctrl+C，SIGTERM kill 命令）
避免服务被外力杀死而不清理资源
5、服务状态可观测
有明确状态机（如 INIT → RUNNING → STOPPING → STOPPED）
便于日志、监控和排查问题
