# 一、线程池

## 1️⃣ 可调用对象

### C 风格函数指针

```cpp
void (*func)(int);
````

* 只能指向“纯函数”
* 只能保存函数地址
* 不能保存对象状态
* 不能绑定成员变量

---

### C++ 风格 std::function

```cpp
std::function<void(int)>
```

* 可以接受几乎所有可调用对象：

  * 普通函数
  * lambda
  * 有捕获的 lambda
  * 仿函数对象
  * 成员函数绑定后的对象

* 可以保存对象和状态

---

### 常见函数签名对应关系

| 函数签名           | std::function 类型       | 调用方式   |
| -------------- | ---------------------- | ------ |
| void f()       | function<void()>       | f()    |
| void f(int)    | function<void(int)>    | f(10)  |
| int f(int,int) | function<int(int,int)> | f(2,3) |

---

## 2️⃣ 线程池执行流程

* 工作线程一开始是阻塞状态
* 通过 condition_variable 等待任务
* 避免 CPU 空转

---

### 提交任务

* 提交任务后：

  * 调用 notify_one 或 notify_all
  * 唤醒线程

---

### 被唤醒线程的流程

1. 先抢锁
2. 检查 predicate 条件

   * 队列非空
   * 或线程池停止
3. 取任务
4. 立即释放锁
5. 在无锁状态下执行任务

---

### 关键点

* 锁只保护任务队列
* 任务执行阶段完全并发
* wait 会临时释放锁
* 唤醒时再重新加锁
* 不会长期阻塞或死锁

---

## 3️⃣ 唤醒机制

### 正常唤醒

* 有线程调用：

  * notify_one()
  * notify_all()
* 操作系统唤醒等待线程

---

### 伪唤醒

* 没有任何 notify
* 条件也未满足
* 线程从 wait 返回

C++ 标准允许伪唤醒，程序必须防范。

---

### 唤醒流程理解

唤醒 ≠ 一定可以工作

流程：

1. 唤醒（门铃响）
2. 加锁
3. 判断 predicate（门锁）

   * true → 继续执行
   * false → 释放锁 → 再次等待

类比：

> 唤醒是门铃响，predicate 是门锁，门锁打开才能进去干活。

---

## 4️⃣ 线程池关闭行为

* 所有任务结束后
* 所有线程进入挂起状态
* 直到主线程调用 shutdown
* 才真正退出

# 二、并发 Bug

## 1️⃣ 数据竞争

* 没有锁
* 没有 atomic
* 多线程同时修改

特点：

* 程序不崩
* 代码不报错
* 结果“不可信”

---

## 2️⃣ 死锁

* 线程完全停住
* 互相等待对方释放资源
* 系统僵死
* CPU 占用可能不高

---

## 3️⃣ 活锁

* 线程一直在动
* 一直 retry
* 逻辑无法推进
* 系统空转

总结：

| 类型 | 状态  | 系统行为 |
| -- | --- | ---- |
| 死锁 | 停住  | 僵死   |
| 活锁 | 一直动 | 无进展  |

你理解为：

> 活锁像是死锁释放资源后循环的版本。

---

## 4️⃣ 数据竞争是否出现

* 取决于：

  * 线程数量
  * CPU 核心数
  * 运行次数
* 核心多时：

  * 可能真正并行
  * 更容易暴露问题
* 运行次数太少：

  * 不容易复现

# 三、I/O 模型实验总结

---

# 一连接一线程模型

特点：

* 线程数随连接数线性增长
* 大量线程阻塞在 read()
* 上下文切换频繁
* CPU 系统态占比高
* 内存和 Swap 压力大
* 扩展性差

---

# select 模型

特点：

* 始终只有 1 个线程
* 统一等待 I/O 事件
* 上下文切换极少
* CPU 更高效
* 内存占用稳定
* 适合高并发 I/O 场景

---

## 核心结论

> 高并发 ≠ 多线程
> I/O 密集场景瓶颈在等待 I/O
> 不是算力
> 减少上下文切换才是关键

---

# 资源使用对比

## 1️⃣ CPU

一连接一线程：

* 线程暴增
* 系统态占比高
* 调度成本高

select：

* 单线程
* 几乎无上下文切换
* 更高效

---

## 2️⃣ 内存

一连接一线程：

* 每线程独立栈
* TCB 巨大
* 易触发 Swap

select：

* 单线程
* 内存增长缓慢
* 几乎不 Swap

---

## 3️⃣ 负载

一连接一线程：

* sleeping 线程多
* Load Average 高

select：

* 线程数恒定
* 系统更健康

---

# 四、poll 模型

## pollfd 结构

```cpp
struct pollfd {
    int   fd;
    short events;
    short revents;
};
```

特点：

* 去除了 fd 数量硬限制
* 每次仍然线性扫描 O(n)
* 高并发低活跃场景下效率不高

---

# 五、epoll 模型

---

## epoll_create1

```cpp
int epoll_create1(int flags);
```

作用：

* 创建 epoll 实例
* 返回 epfd

flags：

* 0
* EPOLL_CLOEXEC

内核中：

* 红黑树（监听集合）
* 就绪队列

---

## epoll_ctl

```cpp
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
```

操作类型：

* EPOLL_CTL_ADD
* EPOLL_CTL_MOD
* EPOLL_CTL_DEL

作用：

* 建立 fd → epoll 映射
* 保存 epoll_event

---

## epoll_wait

```cpp
int nready = epoll_wait(epfd, events.data(), events.size(), -1);
```

参数：

* epfd
* 用户态数组地址
* 数组容量
* 阻塞时间

返回：

* 就绪事件数量

# 六、网络并发模型对比

| 模型     | 数据结构       | 线程数    | 遍历效率 | 适用场景 |
| ------ | ---------- | ------ | ---- | ---- |
| 阻塞     | 无          | 一连接一线程 | -    | 小规模  |
| select | fd_set 位图  | 单线程    | O(n) | 小规模  |
| poll   | pollfd 数组  | 单线程    | O(n) | 中小规模 |
| epoll  | 红黑树 + 就绪队列 | 单线程    | O(1) | 高并发  |

# 八、优雅退出

## 信号

| 信号      | 触发条件       |
| ------- | ---------- |
| SIGINT  | Ctrl+C     |
| SIGTERM | kill -TERM |

---

## 核心目标

优雅退出的本质：

> 停止服务时
> 不丢任务
> 不泄露资源
> 不进入不确定状态

---

### 1️⃣ 执行中的任务完成

* worker 已拿到任务
* 必须执行完
* 不能强杀线程

---

### 2️⃣ 队列任务处理

* 未执行任务：

  * 全部执行
  * 或安全丢弃并记录日志

---

### 3️⃣ 释放资源

* 内存
* 文件句柄
* socket
* mutex
* join 线程
* flush 文件
* close socket

---

### 4️⃣ 响应外部信号

* 正确处理 SIGINT / SIGTERM
* 避免强制 kill 导致资源泄露

---

### 5️⃣ 状态可观测

状态机示例：

```
INIT → RUNNING → STOPPING → STOPPED
```

* 状态明确
* 方便日志
* 方便监控
* 方便排查问题
