# 系统 C++ 阶段二 — 并发与长期运行（21 天）

> 阶段目标：  
> 从「能稳定运行的单服务程序」 → 「可并发、可扩展、可长期运行的系统级服务」  
>
> 核心能力：
> - 并发模型设计
> - 线程池 / 任务模型
> - 网络并发（select / poll / epoll）
> - 长期运行稳定性
> - 工程级可扩展性

---

## 第 1 周：并发基础与线程池

### Day 1 — 并发模型认知
**目标**：建立“并发 ≠ 多线程”的系统认知  
**任务**：
- 梳理并发模型：
  - 单线程事件驱动
  - 多线程阻塞模型
  - Reactor / Proactor
- 回答：
  - 为什么线程多了反而慢？
  - 什么时候不用多线程？
**达标效果**：
- 能口头讲清 3 种并发模型适用场景
- 不再“为了并发而并发”

---

### Day 2 — ThreadPool 骨架
**目标**：抽象任务与执行者  
**任务**：
- 设计 `ThreadPool`：
  - 固定线程数
  - 任务队列
- 任务用 `std::function<void()>`
- 支持安全退出
**达标效果**：
- 线程池能创建、执行任务、优雅退出
- 不发生死锁、野线程

---

### Day 3 — ThreadPool 稳定性
**目标**：线程池异常安全  
**任务**：
- 模拟任务抛异常
- 模拟任务执行超时
- 确保：
  - 线程不退出
  - 服务不崩溃
**达标效果**：
- 线程池长期运行稳定
- 异常被“吃掉并记录”

---

### Day 4 — 任务模型设计
**目标**：从“函数调用”到“任务系统”  
**任务**：
- 区分：
  - IO 任务
  - 计算任务
- 设计 Task 生命周期：
  - 创建 → 入队 → 执行 → 回收
**达标效果**：
- 能清楚画出任务流转图
- 不混用线程职责

---

### Day 5 — ThreadPool 接入 Service
**目标**：并发接入真实服务  
**任务**：
- Service 使用 ThreadPool 处理任务
- main 不直接创建线程
- 模拟高并发任务提交
**达标效果**：
- Service 并发处理能力明显提升
- 架构清晰，不乱

---

### Day 6 — 并发 Bug 实战
**目标**：真正理解并发问题  
**任务**：
- 制造并解决：
  - 数据竞争
  - 死锁
  - 活锁
- 用日志定位问题
**达标效果**：
- 能快速定位并发 Bug
- 不靠“加锁玄学”

---

### Day 7 — 周总结
**目标**：并发基础扎实  
**任务**：
- 总结：
  - 为什么要线程池
  - 线程池设计原则
  - 常见并发坑
**达标效果**：
- 能向别人讲清你的线程池设计

---

## 第 2 周：网络并发模型（重点）

### Day 8 — 阻塞 IO 模型
**目标**：知道“为什么它不行”  
**任务**：
- 写阻塞式 TCP Server
- 一连接一线程
- 压测观察性能
**达标效果**：
- 直观看到线程爆炸问题

---

### Day 9 — select 模型
**目标**：IO 多路复用入门  
**任务**：
- 使用 select 重写 TCP Server
- 支持多连接
- 单线程
**达标效果**：
- 理解 fd 集合扫描
- 能解释 select 的缺陷

---

### Day 10 — poll 模型
**目标**：对比 select / poll  
**任务**：
- 用 poll 重写服务器
- 对比性能与代码复杂度
**达标效果**：
- 知道 poll 改善了什么
- 知道 poll 仍然的问题

---

### Day 11 — epoll 模型（核心）
**目标**：Linux 网络并发核心  
**任务**：
- 使用 epoll 实现 TCP Server
- 支持多客户端并发
- 水平触发（LT）
**达标效果**：
- 稳定处理多连接
- 能画出 epoll 事件流程

---

### Day 12 — epoll + ThreadPool
**目标**：工业级并发模型  
**任务**：
- epoll 只负责 IO
- 业务逻辑交给 ThreadPool
- 防止阻塞 epoll 线程
**达标效果**：
- epoll 线程永不阻塞
- 架构清晰

---

### Day 13 — 网络异常处理
**目标**：真实网络环境  
**任务**：
- 处理：
  - 客户端异常断开
  - 半包 / 粘包
- 连接正确回收
**达标效果**：
- 不泄露 fd
- 服务长期稳定

---

### Day 14 — 网络并发总结
**目标**：真正“懂网络”  
**任务**：
- 对比：
  - 阻塞 / select / poll / epoll
- 写总结文档
**达标效果**：
- 面试可完整讲清网络并发模型

---

## 第 3 周：长期运行与工程化

### Day 15 — 服务状态机
**目标**：服务生命周期可控

### Day 20 — 整体重构
