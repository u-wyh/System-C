
# C++ 系统资源管理笔记（完整版）

---

## 1️⃣ 文件类 File / FILE*

### FILE* 的特点
- **C 标准库的文件流指针**  
- **不是普通内存**，而是 **内核 + 用户态联合管理** 的资源
- 内部结构：
  - 文件描述符 `fd`（内核分配）
  - 缓冲区 `buffer`（用户态管理）
  - 文件位置指针（当前读写位置）
  - 状态标志（EOF、错误标志）

### 深拷贝问题
- **无法实现深拷贝 FILE***  
- File 对象的拷贝不是数据拷贝，而是 **资源所有权拷贝**
- **资源只能有一个主人**  

### 移动操作 (move)
- 原资源全部转移
- 原指针指向 `nullptr`，不是野指针
- 可以手动实现 move（在 File 类中可自定义移动构造和移动赋值）

### 写操作
- `write` 并不是直接写入内存，而是写进缓冲区  
- 优化性能，减少系统调用次数

---

## 2️⃣ Socket 类

- **系统调用函数前加 `::`**  
- 内部也使用内核资源（文件描述符）
- 类似 File 类，资源需要唯一所有权管理
- 可以使用移动操作转移所有权

---

## 3️⃣ Mutex 类

### 基本概念
- **解决竞争问题**：锁与解锁保护共享资源  
- **临界区**：访问共享资源的代码段
  - 临界区本身不是锁
  - 上锁 → 线程才能进入
  - 解锁 → 其他线程可以进入

### futex
- 高性能锁机制，避免频繁系统调用  
- 大部分时候线程没有竞争，频繁上锁解锁会影响性能

### LockGuard
- **RAII 管理 mutex 生命周期**
  - 构造 → 上锁  
  - 析构 → 解锁
- **优点**：
  - 自动管理生命周期
  - 减少忘记解锁的风险

### unique_lock vs lock_guard
| 特性 | lock_guard | unique_lock |
|------|------------|-------------|
| 灵活性 | 不可 unlock/relock | 可以 unlock/relock |
| 使用场景 | 简单加锁 | 条件变量、复杂场景 |
| 轻量性 | 轻量 | 相对重一点 |

## 4️⃣ 线程 thread 类

### 基本概念
- **并发执行**，抢占式调度
- `join`：阻塞式，等待线程执行完  
- `detach`：非阻塞式，主线程继续执行，结束时线程终止

### 可变参数包（ellipsis `...`）
- 用于模板或函数参数声明
- 可展开参数包传给实际函数
- 作用：
  - 模板声明：可变数量类型参数
  - 函数参数：可变数量函数参数
  - 调用时展开：拆开参数包传递

### 类比理解
- **File（你是房主）**
  - 房子：`FILE*`
  - 你知道钥匙在哪里
  - 你可以手写 move
- **std::thread（你是租客）**
  - 房子：内核线程
  - 房东：操作系统
  - 中介：`std::thread`
  - 你只能让中介转租，不能自己撬门换锁

## 5️⃣ 综合操作注意事项

### 1. 类成员初始化限制
```cpp
// ❌ 错误示例
static File f("a.txt","w"); // 类成员声明处禁止圆括号初始化
````

### 2. 多线程共享变量

* 使用 `atomic`，保证原子操作
* 原子操作 = 不可打断 + 不可重排 + 对所有线程立即可见
* 避免类似 `count++` 的并发问题

```cpp
// count++ 实际操作：
// 1. 读取 count
// 2. count + 1
// 3. 写回 count
// 多线程同时执行可能导致冲突
// atomic 可以保证三步作为一个不可分割操作
std::atomic<int> count;
count++;
```

### 3. 临界区与锁

* 临界区是一段访问共享资源的代码
* 锁只是保护工具
* 上锁 → 线程进入临界区
* 解锁 → 其他线程可以进入

## 7️⃣ 总结

* **File / FILE***：资源唯一所有权 → 移动，不能拷贝
* **Socket**：类似 File，需要管理内核资源
* **Mutex / LockGuard**：保护临界区，避免竞争
* **thread**：并发执行，需要 join/detach 管理线程生命周期
* **atomic**：保证多线程共享变量安全

> 核心思想：C++ 系统资源管理的本质是 **资源所有权**，谁拥有资源，谁负责释放，移动资源而非拷贝资源，保证线程安全与性能。
