文件类：

FILE* 是 C 标准库的文件流指针，特点：

它不是普通内存，而是 内核 + 用户态联合管理的资源

内部有：

文件描述符 fd（内核分配）

缓冲区 buffer（用户态管理）

文件位置指针（当前读写位置）

状态标志（EOF、错误标志）

所以深拷贝 FILE* 本质上是 无法实现 的。FILE类只能移动，不能拷贝
File 对象的拷贝不是数据拷贝，而是资源所有权拷贝，资源只能有一个主人

移动move操作之后，原来的资源全部转移，原指针指向nullptr，不是野指针

write操作不是直接写入内存，而是写进缓冲区，这样可以优化性能

Mutex类：

涉及到了futex。
mutex是为了解决竞争问题，具体方法是锁与解锁，已经使用的数据，锁上；不用了，就解锁。
但是大部分是没有竞争的，如果频繁的锁和解锁，就会影响性能。而且有竞争的，也必须要处理。
因此，引入了futex。

LockGuard是用于管理mutex对象的生命周期，构造时加锁，析构时解锁

这里之所以不将mutex类和lockGuard写在一起，是因为有的mutex对象本身是全局变量，构造时就上锁，灵活性差。
全局/长期 mutex 无法控制局部临界区。 多线程共享 mutex 时容易出错。

什么是临界区：是一段 访问共享资源的代码，在多线程环境下 同一时刻只能被一个线程执行。临界区本身不是锁，锁只是保护它的工具。上锁 → 线程才能进入临界区，解锁 → 其他线程可以进入。可以简单的认为就是上锁到解锁的中间的那部分代码。

线程的join是阻塞式，就是说必须要执行完改线程之后才能执行主进程
detch是非阻塞式，主进程照常进行，主进程结束时，线程立即终止
线程是并发进行的，代码中的两个是抢占式的

unique_lock:灵活，可以 unlock/relock，用在条件变量等复杂场景
lock_guard:轻量、简单、只管理生命周期

综合操作：
在service类的静态成员中不能File f("a.txt","w");  C++ 语法明确禁止在“类成员声明处”使用圆括号 () 进行初始化。
在多线程中共享一个变量的时候，需要用atomic，比如这个程序中，如果使用bool变量的话，那么对于每个线程而言，他会发现自己从来没有改过这个值，那么就按照最开始的值来用，这样就可能导致问题。而atomic保证每次都是原子操作。
原子操作 = 不可打断 + 不可重排 + 对所有线程立即可见。非原子操作可能会导致的问题，可以参考mutex文件中注释掉63行后的效果。
比如count++，这一步实际上是  读取count + count+1 + 写回，多线程的时候，并发执行，可能存在问题。原子操作就是这三步一起执行，对于某个线程而言，要么不执行，要么执行全部，不存在执行一部分，然后另一个线程执行了。



状态安全性：
状态 commit = 只有不可失败点成功后才修改
以本程序为例，如果程序中的部分线程构造失败，那么running已经被设置为了true，这个不会改变。在本程序中实际是没有什么影响的，但是如果是系统中的话，那么可能会影响很大。
任何状态错误，都会被线程读取 → 导致“半启动 / 错误运行”
