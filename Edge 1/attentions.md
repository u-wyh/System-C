
# C++ 系统编程：资源、并发与状态安全笔记

---

## 一、文件类（File / FILE*）

### FILE* 的本质
- `FILE*` 是 **C 标准库的文件流指针**
- **不是普通内存**
- 是 **内核 + 用户态联合管理的资源**

### FILE* 内部包含
- 文件描述符 `fd`（内核分配）
- 缓冲区 `buffer`（用户态管理）
- 文件位置指针（当前读写位置）
- 状态标志（EOF、错误标志）

### 拷贝与移动
- **深拷贝 `FILE*` 本质上是无法实现的**
- `File` 类 **只能移动，不能拷贝**
- `File` 对象的拷贝不是数据拷贝，而是**资源所有权拷贝**
- **资源只能有一个主人**

### move 语义
- move 之后：
  - 原资源全部转移
  - 原指针指向 `nullptr`
  - **不是野指针**

### write 行为
- `write` 操作不是直接写入内存
- 而是写入 **缓冲区**
- 这样可以减少系统调用，提高性能

## 三、Mutex 类

### mutex 的作用

* mutex 用来解决 **竞争问题**
* 基本方式：

  * 数据正在使用 → 上锁
  * 数据不用了 → 解锁

### 性能问题

* 大部分情况下 **没有竞争**
* 如果频繁 lock / unlock：

  * 会影响性能
* 但一旦有竞争：

  * 又必须正确处理

### futex 的引入

* 为了解决：

  * 无竞争时的性能问题
  * 有竞争时的正确性问题
* Linux 使用 **futex** 作为底层机制

---

### LockGuard

* `LockGuard` 用于管理 `mutex` 的生命周期
* **RAII 思想**

  * 构造时加锁
  * 析构时解锁

### 为什么 mutex 和 LockGuard 不写在一起

* 有些 `mutex` 是：

  * 全局变量
  * 长生命周期对象
* 如果构造时就上锁：

  * 灵活性很差
* **全局 / 长期 mutex**

  * 无法精确控制局部临界区
  * 多线程共享时容易出错

---

### 什么是临界区

* 临界区是：

  * **访问共享资源的一段代码**
* 多线程环境下：

  * 同一时刻只能被一个线程执行
* 临界区本身 **不是锁**
* 锁只是保护临界区的工具

流程可以理解为：

```
lock
  ↓
临界区代码
  ↓
unlock
```

---

### unique_lock vs lock_guard

* `unique_lock`

  * 更灵活
  * 可以 `unlock / relock`
  * 常用于条件变量等复杂场景

* `lock_guard`

  * 轻量
  * 简单
  * 只负责生命周期管理

## 四、thread 类

### join / detach

* `join`

  * 阻塞式
  * 必须等待线程执行完
  * 主线程才继续

* `detach`

  * 非阻塞式
  * 主线程照常运行
  * 主线程结束时，线程立即终止

### 线程模型

* 线程是 **并发执行**
* 调度是 **抢占式**

## 七、综合操作注意事项

### 1. 类静态成员初始化问题

在 `Service` 类的静态成员中：

```cpp
File f("a.txt","w"); // ❌
```

这是非法的。

原因：

* C++ 明确禁止在 **类成员声明处**
* 使用 **圆括号 `()` 进行初始化**

---

### 2. 多线程共享变量

* 多线程中共享变量：

  * 必须使用 `atomic`
* 如果使用普通 `bool`：

  * 每个线程可能读取到的是旧值
  * 认为自己从未修改过
  * 导致严重逻辑错误

### atomic 的保证

* 原子操作 =

  * 不可打断
  * 不可重排
  * 对所有线程立即可见

---

### 3. 非原子操作的问题

例如：

```cpp
count++;
```

实际上分三步：

1. 读取 `count`
2. `count + 1`
3. 写回 `count`

多线程并发执行时：

* 可能发生交叉
* 造成数据错误

原子操作保证：

* 要么不执行
* 要么三步全部完成
* 不存在执行一半被打断

## 八、状态安全性（State Safety）

### 状态提交原则

* **状态 commit**

  * 只有在不可失败点成功后
  * 才允许修改状态

### 本程序中的问题

* 如果部分线程构造失败
* `running` 已经被设置为 `true`
* 这个状态不会再回滚

在当前程序中：

* 影响不明显

但在真实系统中：

* 后果可能非常严重

### 风险

* 任何状态错误：

  * 都会被线程读取
  * 导致：

    * 半启动
    * 错误运行

## 九、持久化运行问题（Day9 程序）

### 存在的问题

* `day9` 程序存在多个严重隐患

#### 1. running 状态问题

* `running` 的修改方式是错误的
* 线程中存在：

```cpp
while (running)
```

如果状态设置不当：

* 程序可能一启动就立即退出

#### 2. 资源泄露问题

* `tcp_server` 函数中
* 可能导致 `client_fd` 泄露
* **这是非常严重的系统级错误**
